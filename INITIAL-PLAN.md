## Hackathon PoC

### 1. BDD Input

A very small scenario file, even if it is semi-structured. Example:

```
Scenario: Admin enrolls learner

Given I am logged in as an admin
When I open the course "Security Training"
And I enroll "John Smith"
Then I should see "John Smith" in the roster
```

For the hackathon, this does not need a strict vocabulary. It just needs to be understandable enough for the planner.

---

### 2. App Explorer

This is the most important part. An AI should connect to a real app instance and collect:

- current routes
- page title / headings
- visible buttons
- form labels
- inputs
- links
- candidate selectors
- maybe screenshots
- maybe DOM snapshots

Output should be a lightweight JSON page model like:

```json
{
  "page": "course_details",
  "url": "/courses/123",
  "elements": [
    {
      "type": "button",
      "text": "Enroll users",
      "selector": "[data-testid='enroll-users']"
    },
    {
      "type": "input",
      "label": "Search users",
      "selector": "[data-testid='user-search']"
    }
  ]
}
```

This is the part that proves "AI can understand the actual app."

---

### 3. Planner

This takes:
- the BDD
- the app explorer output

and produces a structured action plan. Example:

```json
{
  "scenario": "Admin enrolls learner",
  "steps": [
    { "action": "login", "role": "admin" },
    { "action": "navigate", "target": "course_details", "course": "Security Training" },
    { "action": "click", "target": "Enroll users" },
    { "action": "fill", "target": "Search users", "value": "John Smith" },
    { "action": "select", "target": "John Smith" },
    { "action": "assert_visible", "target": "roster", "value": "John Smith" }
  ]
}
```

This is where AI should do most of its work.

---

### 4. Code Generator

This converts the plan into Playwright code. Example output:

```typescript
test("Admin enrolls learner", async ({ page }) => {
  await page.goto("/login");
  await page.fill('input[name="username"]', "admin");
  await page.fill('input[name="password"]', "password");
  await page.click('button:has-text("Login")');

  await page.goto("/courses/123");
  await page.click('[data-testid="enroll-users"]');
  await page.fill('[data-testid="user-search"]', "John Smith");
  await page.click(`text=John Smith`);

  await expect(page.locator('[data-testid="roster"]')).toContainText("John Smith");
});
```

This does not need to be perfect. It just needs to be credible.

---

### Optional Fifth Block: Basic Execution

I would still include very minimal execution, but not as a major investment.

Reason: generated code alone is interesting, but generated code that actually runs — even on one happy path — is much more convincing.

So execution should be:
- run one generated Playwright spec
- show pass or partial failure
- collect screenshot / trace

That is enough.

---

### Best Hackathon Demo Story

The strongest demo is:

1. Show a BDD scenario.
2. Point the system at a real app instance.
3. Let the explorer inspect the page(s).
4. Show the generated plan.
5. Show the generated Playwright code.
6. Run it.

That tells a complete story.

---

### What "App Explorer" Should Really Do in the Hackathon

Keep it narrower than a full crawler. Do not try to discover the whole application. Instead, make it a targeted explorer:

- start from a known URL
- inspect the current page
- maybe follow one or two interactions
- extract interactable elements and selectors
- return structured metadata

So rather than "explore the whole app," it becomes: "inspect the relevant pages well enough to support one generated test." That is much more achievable in two days.

---

### Recommended Scope for the App Explorer

In the hackathon, it should support only:
- buttons
- inputs
- links
- text assertions
- route changes

Ignore for now:
- drag and drop
- iframes
- file uploads
- rich editors
- complex widgets

---

### The Simplest Workable Architecture

```
scenario.feature
   -> BDD parser
   -> LLM planner
      + page metadata from explorer
   -> plan.json
   -> codegen template
   -> generated.spec.ts
   -> playwright test
```

---

### What Can Be Mocked or Hardcoded

To stay within two days, you should hardcode aggressively:
- credentials
- one test environment
- one scenario family
- one or two routes
- one code template
- one browser
- one app area

That is fine. The point is not generality. The point is proof.

## Full Platform Architecture & Building Blocks

### Overview

This document outlines the main building blocks required to implement an AI-assisted QA platform based on:

- Fixed vocabulary BDD scenarios
- Automated test generation
- Application exploration
- Test execution
- Continuous improvement loops

**High-level pipeline:**

```
Behavior Specification
      ↓
Quality & Governance Checks
      ↓
Application Understanding
      ↓
Test Planning
      ↓
Code Generation
      ↓
Test Execution
      ↓
Failure Analysis
      ↓
Continuous Improvement
```

---

### 1. Vocabulary Registry

**Purpose:** The source of truth for the fixed testing vocabulary.

**Responsibilities — defines:**
- Allowed actions
- Allowed assertions
- Context setters
- Parameter patterns
- Canonical phrasing
- Deprecated phrases

**Example**

Canonical term: `login as {role}`

Non-canonical variants prevented:
- sign in
- authenticate
- log into system

**Outputs:**
- Versioned vocabulary schema
- Documentation
- Machine-readable definitions

---

### 2. BDD Authoring Layer

**Purpose:** Human interface where QA engineers define behavior scenarios.

Possible formats:
- Gherkin
- Structured Markdown
- Custom editor with autocomplete

**Example:**

```
Scenario: Admin enrolls learner

Given login as admin
When open course "Security Training"
And enroll user "John Smith"
Then user enrolled in course
```

**Responsibilities:**
- Capture product behavior
- Maintain readability
- Enforce scenario structure

---

### 3. Vocabulary Linter / Policy Validator

**Purpose:** Deterministic validation layer.

**Checks:**
- Approved vocabulary usage
- Grammar pattern compliance
- Required parameters
- Banned phrases
- Multiple actions per step

**Example Flags:**
```
Given I sign in as admin
When click the blue button
Then it works
```

**Outputs:**
- Validation result
- Exact violations
- Suggested canonical replacements

---

### 4. AI Vocabulary Reviewer

**Purpose:** Semantic review layer on top of deterministic rules.

**Evaluates:**
- Synonyms
- Ambiguity
- UI leakage
- Poor abstraction
- Missing vocabulary concepts

**Example**

Input: `When I click enroll users and search John`

Suggested rewrite: `When open enrollment panel` / `And search user "John"`

**Outputs:**
- Warnings
- Rewrite suggestions
- Vocabulary expansion proposals

---

### 5. Vocabulary Governance Module

**Purpose:** Manages the lifecycle of the vocabulary.

**Responsibilities:**
- Approve new terms
- Deprecate old terms
- Review repeated deviations
- Identify overlapping commands

**Example:** Repeated deviations lead to proposed command: `impersonate learner "{user}"`

---

### 6. Scenario Parser

**Purpose:** Convert validated BDD into structured machine format.

**Example Output:**

```json
{
  "scenario": "Admin enrolls learner",
  "steps": [
    {"type": "action", "name": "login", "params": {"role": "admin"}},
    {"type": "action", "name": "open_course", "params": {"course": "Security Training"}},
    {"type": "action", "name": "enroll_user", "params": {"user": "John Smith"}},
    {"type": "assertion", "name": "user_enrolled", "params": {"user": "John Smith"}}
  ]
}
```

---

### 7. Application Knowledge / Exploration Engine

**Purpose:** Build a structured understanding of the running application.

**Collects:**
- Routes
- Screens/pages
- UI elements
- Accessible labels
- Stable selectors
- Component patterns
- Network activity
- Authentication flows

**Output:**
- Page model
- Selector catalog
- Action metadata

---

### 8. Test Data & Environment Manager

**Purpose:** Prepare and maintain stable environments for testing.

**Responsibilities:**
- Data seeding
- Fixture management
- Environment provisioning
- State reset
- Authentication bootstrap

**Outputs:**
- Test-ready environment
- Seeded datasets

---

### 9. Action Library

**Purpose:** Reusable automation primitives mapped to vocabulary commands.

**Example:**
```
loginAs(role)
openCourse(name)
enrollUser(user)
expectUserEnrolled(user)
```

**Benefit:** Avoids AI generating low-level browser interactions repeatedly.

---

### 10. Planning Engine

**Purpose:** Translate parsed scenarios into executable test plans.

**Responsibilities:**
- Determine setup requirements
- Map vocabulary to automation actions
- Decide between UI or API usage
- Assemble execution sequence

**Example:**

```json
{
  "setup": [
    {"action": "seed_course", "course": "Security Training"}
  ],
  "execution": [
    {"action": "loginAs", "args": ["admin"]},
    {"action": "openCourse", "args": ["Security Training"]},
    {"action": "enrollUser", "args": ["John Smith"]}
  ],
  "assertions": [
    {"action": "expectUserEnrolled", "args": ["John Smith"]}
  ]
}
```

---

### 11. AI Planner / Reasoning Layer

**Purpose:** Assist when deterministic mappings are insufficient.

**Capabilities:**
- Map intent to actions
- Handle partial mappings
- Suggest new action compositions

**Guardrail:** AI should be used after deterministic logic, not before.

---

### 12. Code Generator

**Purpose:** Generate executable automation tests.

**Targets:**
- Playwright tests
- Cypress tests

**Responsibilities:**
- Follow coding conventions
- Use stable selectors
- Prefer reusable actions

---

### 13. Execution Runner

**Purpose:** Run generated tests in target environments.

**Handles:**
- Local execution
- CI pipelines
- Parallelization
- Browser matrix

**Outputs:**
- Pass/fail status
- Logs
- Screenshots
- Videos/traces

---

### 14. Failure Analyzer

**Purpose:** Diagnose failed test runs.

**Failure Categories:**
- Selector drift
- Timing issues
- Bad test generation
- Ambiguous BDD
- Missing test data
- Product defect

**Outputs:**
- Root cause classification
- Suggested fixes

---

### 15. Healing / Repair Engine

**Purpose:** Suggest fixes for failed tests.

**Possible Actions:**
- Update selectors
- Add waits or assertions
- Rewrite scenarios
- Expand vocabulary

**Guardrail:** Prefer suggesting fixes before automatic mutation.

---

### 16. Review & Approval Workflow

**Purpose:** Human oversight for critical decisions.

**Used For:**
- Reviewing rewritten scenarios
- Approving vocabulary changes
- Accepting test healing suggestions

---

### 17. Reporting & Observability

**Purpose:** Provide insight into system performance.

**Metrics:**
- Scenario quality
- Vocabulary violations
- Pass/fail rate
- Flakiness
- Failure causes
- Coverage of vocabulary actions

---

### 18. Feedback & Learning Loop

**Purpose:** Continuously improve the system.

**Uses Data From:**
- Authoring deviations
- Frequent failures
- Manual test edits
- Healing outcomes

**Improves:**
- Vocabulary
- Action library
- Planner
- Code generator

---

### 19. Security & Access Control

**Purpose:** Protect environments and credentials.

**Handles:**
- Secrets management
- Role-based test accounts
- API tokens
- Data masking

---

### 20. Developer / QA Experience Layer

**Purpose:** Provide a usable interface for teams.

**Features:**
- Scenario editor
- Autocomplete vocabulary
- Inline linting
- Rewrite suggestions
- Test generation controls
